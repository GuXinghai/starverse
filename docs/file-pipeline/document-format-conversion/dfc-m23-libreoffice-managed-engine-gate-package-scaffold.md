# DFC-M23 LibreOffice Managed Engine Gate / Package Scaffold

Date: 2026-06-01
Branch: docs/dfc-0-format-conversion-foundation
Baseline: f0a62e5

## Scope

M23 adds a LibreOffice managed runtime availability gate for the future DOCX-first Office-to-PDF path. It is a package/manifest scaffold and DFC candidate exposure slice only.

M23 does not bundle LibreOffice, does not call `soffice`, does not use system LibreOffice, does not generate PDFs, and does not declare production Office-to-PDF support.

## Package scaffold contract

The managed runtime gate expects a Starverse-owned LibreOffice package rooted under the app-managed runtime area:

- package id: `starverse.dfc.libreoffice`
- engine id: `libreoffice`
- runtime id: `libreoffice-office-pdf`
- platform and optional arch
- executable relative path, such as `program/soffice` or `program/soffice.exe`
- LibreOffice version
- package version
- artifact SHA-256
- executable SHA-256
- executable size
- provenance
- license id
- notices metadata
- capabilities: `office_to_pdf` and `docx_to_pdf`
- minimum Starverse contract version
- security policy metadata:
  - macros disabled
  - network disabled
  - external links disabled
  - isolated profile required

The gate returns a renderer-safe summary only. It does not return executable paths, package root paths, hash values, manifest body, notices body, command/env, or license body through DFC DTOs.

## Availability diagnostics

Symbolic diagnostics:

- `office_pdf_runtime_missing`
- `office_pdf_runtime_manifest_invalid`
- `office_pdf_runtime_executable_missing`
- `office_pdf_runtime_path_rejected`
- `office_pdf_runtime_platform_unsupported`
- `office_pdf_runtime_metadata_incomplete`

Path policy:

- reject absolute paths
- reject UNC paths
- reject drive escapes
- reject path traversal
- reject NUL
- require executable path to resolve under the managed runtime root

Metadata policy:

- missing hash, size, provenance, license, notices, required capabilities, or required security policy produces `office_pdf_runtime_metadata_incomplete`
- executable size/hash mismatch produces `office_pdf_runtime_manifest_invalid`

## DFC behavior

DOCX draft attachments can now expose an unavailable/blocked `pdf_attachment` candidate when the managed LibreOffice runtime is missing or invalid.

Current semantics:

- `original_file` remains ready and independent.
- DOCX `markdown` remains ready through the existing Mammoth pilot.
- DOCX `pdf_attachment` is blocked/unavailable until a real managed LibreOffice package and conversion adapter are approved.
- Unavailable PDF candidates have no `sendAssetRefs`.
- Unavailable PDF candidates cannot become valid persisted selected options.
- No ready `converted_pdf` DerivedAsset is generated by M23.
- `.doc`, `.rtf`, and `.docm` remain unsupported and do not expose Office-to-PDF candidates.
- No legacy fallback is added.

## Non-goals

- No LibreOffice binary.
- No `soffice` execution.
- No system LibreOffice fallback.
- No Office-to-PDF conversion generation.
- No DOC, RTF, or DOCM support.
- No DB schema, renderer IPC shape, Send Plan main-flow, asset model, or DFC vocabulary change.
- No HTML-to-PDF pipeline change.
- No packaged installer, CI, full-suite failure cleanup, npm audit, or ESLint work.
