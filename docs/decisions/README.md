# 架构决策记录 (Architecture Decision Records)

本目录记录 Starverse 项目的重大技术决策，帮助团队理解"为什么这样做"。

## 📖 什么是 ADR？

ADR（Architecture Decision Record）是一种轻量级文档格式，用于记录软件架构中的重要决策及其背景。

### ADR 的价值

- ✅ **避免重复讨论**: 新成员可快速了解历史决策
- ✅ **决策透明化**: 记录决策的权衡和背景
- ✅ **知识传承**: 核心技术选型不因人员变动而遗失
- ✅ **责任明确**: 每个决策有明确的提出者和日期

## 📝 ADR 模板

新的 ADR 应包含以下部分：

```markdown
# ADR-XXX: [决策标题]

**状态**: 已提议 / 已接受 / 已废弃 / 已替代  
**日期**: YYYY-MM-DD  
**决策者**: @用户名

## 背景

描述需要做出决策的背景和问题。

## 决策

简明扼要地说明做出的决策。

## 理由

详细解释为什么做出这个决策，包括：
- 考虑了哪些替代方案
- 每个方案的优缺点
- 为什么选择当前方案

## 后果

描述这个决策带来的影响：
- 积极影响
- 消极影响
- 需要采取的行动

## 参考资料

相关链接、文档或讨论记录。
```

## 🔍 如何使用 ADR

### 何时创建 ADR？

当你面临以下情况时，应该创建 ADR：
- ✅ 选择核心技术栈（框架、语言、数据库等）
- ✅ 设计关键架构模式（三层架构、微服务等）
- ✅ 引入重大依赖或工具
- ✅ 做出影响整体系统的设计决策
- ❌ 不需要为小的实现细节创建 ADR

### 创建流程

1. 复制 ADR 模板
2. 编号为下一个可用编号（如 ADR-006）
3. 填写所有必要字段
4. 提交 PR 并邀请团队审查
5. 审查通过后合并并更新本索引

### 状态说明

- **已提议 (Proposed)**: 决策尚在讨论中
- **已接受 (Accepted)**: 决策已被团队接受并实施
- **已废弃 (Deprecated)**: 决策已过时但仍有历史价值
- **已替代 (Superseded)**: 决策被新的 ADR 替代
